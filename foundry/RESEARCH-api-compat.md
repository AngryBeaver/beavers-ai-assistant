Generated by research agent. Module targets `minimum: "13", verified: "13"`.

## Detail

### `ApplicationV2`

Safe in v13. Safe in v14 with one confirmed breaking rename.

- The existing `ApiSettings.ts` already uses `ApplicationV2` correctly via `foundry.applications.api.ApplicationV2` — use this as the reference pattern for `AiGmWindow`.
- **v14 breaking change**: `_replaceHTML` is renamed to `_replaceContent`. Signature changes to `_replaceContent(result, content, options)`. This affects the existing `ApiSettings.ts` as well as the new panel. Not a blocker for v13 shipping, but must be fixed before bumping `verified` to `14`.
- For streaming token updates in the panel: directly mutate the DOM via `this.element` — do not call `render()` on every token.
- `ApplicationV2` does not fire legacy `render<AppName>` hooks from v1. Self-contained panels are unaffected.

### `Folder.create({ name, type: 'JournalEntry' })`

Unchanged. The existing `JournalApi.ts:68` uses this exact call. Safe.

### `JournalEntry.create({ name, folder })`

Unchanged. The existing `JournalApi.ts:84` uses this. Safe.

### `Actor.create({ type: 'npc' })`

`Actor.create()` is stable. The `type` field is **system-defined**, not a Foundry core value. `'npc'` is correct for dnd5e and most fantasy systems but breaks in others (e.g. Warhammer uses `'creature'`).

Options:
- Hardcode `'npc'` and document the system dependency
- Make actor type a module setting
- Use `game.system.documentTypes.Actor[0]` as a safe fallback (first available type in whatever system is running)

For a module that is purely using actors as data stores (flags only), the type matters less — any valid type works.

### `actor.flags` / `actor.update(flags)`

Flag API is the most stable part of Foundry. Recommended patterns:

- **Read**: `actor.getFlag('beavers-ai-assistant', 'key')` — preferred over direct property access
- **Write single key**: `actor.setFlag('beavers-ai-assistant', 'key', value)`
- **Write whole schema**: `actor.update({ 'flags.beavers-ai-assistant.key': value })` using dot notation for nested writes
- Avoid `actor.update({ flags: { 'beavers-ai-assistant': { ... } } })` for partial updates — it merges shallowly and can leave stale nested keys

### `socketlib`

Still the correct approach in v13 and v14. Foundry has not introduced a higher-level native RPC mechanism. The existing `Hooks.once('socketlib.ready', ...)` + `socketlib.registerModule` + `socket.register` pattern is unchanged.

**Important**: use `socket.executeAsGM(methodName, ...args)` for journal writes and actor creates that must run with GM privileges. Confirm all `writeSessionData` and persona-save calls use `executeAsGM`, not `executeForEveryone`.

### `game.scenes.active`

Safe. Returns the active `Scene` document or `null`.

**Critical distinction for `ContextBuilder.ts`:**
- `scene.notes` = collection of map pin `Note` documents (NOT the GM text)
- `scene.description` = the GM-facing freeform description/notes (HTML string) — **this is what ContextBuilder should read**

### `game.journal` / `game.actors`

Both `WorldCollection` instances. Standard methods (`get`, `getName`, `find`, `filter`, `contents`) unchanged in v13 and v14.

### `Hooks.once('socketlib.ready', ...)`

Fires after `init`, before `ready`. Unchanged. Existing registration pattern in `beavers-ai-assistant.ts` is correct.

## Critical Items for Implementation

1. **`scene.description` not `scene.notes`** — silent context failure if wrong property used in `ContextBuilder.ts`
2. **`_replaceHTML` → `_replaceContent`** — must fix in `ApiSettings.ts` AND `AiGmWindow.ts` before v14 support
3. **`executeAsGM`** — all privileged socket calls (journal write, actor create/update) must use this
4. **Actor type** — decide before implementing `PersonaResolver.ts`: hardcode `'npc'` or make configurable
